# CC算法开发-v10 PPT 修改建议

审阅对象：`D:/Claude/Ohmybrain/draft/CC算法开发-v10.pptx`

审阅日期：2026-06-26

## 一句话总评

这份 PPT 已经有清晰的 fieldbook 气质：章节完整、视觉统一、案例证据充分。当前最主要的问题不是“页面不好看”，而是主线被大量工具清单、反模式清单和成果展陈稀释。建议把它分成两层：主线汇报 25-30 页，完整资料/工具箱放附录。

## 优先级最高的修改

### 1. 把第 4 页变成全 deck 的总论点页

第 4 页“Before / Now / Result”是整份 PPT 的核心观点，建议进一步强化。

当前表达：

> 中间四步拆给 agent，研究者只保留最后一步

建议改成：

> AI 没有替代算法研究者，而是把中间四步标准化；研究者的价值上移到判断、边界和方向决策。

建议处理：

- 保留 Before / Now / Result 三段结构。
- 把底部大结论作为全 deck 的主命题。
- 后续每个章节都回扣“中间四步可拆，判断不可拆”。

### 2. 前移第 32 页“知识闭环：流程视图”

第 32 页 `ingest -> query -> promote -> review` 是整套工作流最清楚的地图，但现在出现太晚。

建议：

- 放到第 7 页“三大 AI 能力”之后。
- 作为全篇导航页，告诉观众后面每章分别解释闭环中的哪个动作。
- 第 33 页“三命令”保留为操作页或附录页，不要和第 32 页承担同一个叙事功能。

### 3. 压缩重复的“挑战 -> 解药”页面

第 14、18、26、31 页都是“挑战 -> 解药”结构。单页都清楚，但连续出现会让汇报节奏变成清单。

建议：

- 主线只保留 1-2 张最关键的“挑战 -> 解药”页。
- 其余页面改成案例页，或移到附录。
- 如果保留，建议统一排序维度：严重性、频率、影响阶段。

### 4. 修正第 28 页的证据呈现

第 28 页标题是“CASE 01 · 单根因审计 · 43% -> 0%”，但右侧图目前看起来只展示了修复前约 50% 的曲线。

风险：

- 观众会短暂觉得“标题说 0%，图上却还在 50% 左右”。
- 会削弱这个强案例的可信度。

建议：

- 改成 before / after 双图。
- 或在同一张图上补 after 曲线。
- 或明确标注右侧图是“诊断前/开关前曲线”，再补一张“修复后 0%”的小结果图。

### 5. 放大第 20、21 页的数据证据

第 20 页“V->V->V 递进 · PMF 双指标”和第 21 页“算法 fix 收益快照”是最能证明方法有效的证据页。

建议：

- 第 20 页右上曲线加 1-2 个箭头标注，解释为什么它说明“诚实递进，含未过版”。
- 第 21 页减少 6 张小图的细节压力，保留 1-2 张代表性大图。
- 右侧收益清单可以保留完整，但数字层级要更突出。

### 6. 将第 37、38、40、47 页转为附录/工具箱

这些页信息很有价值，但更像资料清单：

- 第 37 页：工具栈一览
- 第 38 页：反模式合并视图
- 第 40 页：多项目交付 + 专利预研
- 第 47 页：模板、skill 与协作骨架清单

建议：

- 主线只保留最能支撑观点的 1 张。
- 详细清单放 Appendix。
- 结尾不要被工具清单占据，应该回到“新工作流如何改变算法研究”。

## 页面级建议

优点：

- Before / Now / Result 对比自然。
- 适合承担全 deck 的主命题。

建议：

- 把“研究者只保留最后一步”改成更具判断力的表达。
- 底部大结论可以更短、更狠。
- Result 中的“6 体制 · 13 模块 · 186 函数”建议加一句“这是流程产物，不只是代码产物”。

### 第 7 页：精益研究的三大 AI 能力

优点：

- 三张大数字卡很有记忆点。
- 是很好的路线图页。

建议：

- 后续章节标题尽量回扣这三项能力。
- 三个数字可以对应三种收益：调研吞吐、验证收益、知识资产。

### 第 11 页：Hub 大脑功能 · 8 类信息

- 其余条目移到附录。

## 版式与表达建议

### 1. 让 Takeaway 更像“结论”，少像“页脚”

目前多数页底部都有 Takeaway，但视觉上偏页脚。建议主线页把 Takeaway 做得更突出，尤其是第 4、20、21、23、27、28 页。

### 2. 对内部术语做第一次定义

以下术语建议在第一次出现时给一句定义：

- V->V->V
- PMF 双指标
- handoff
- promote
- Hub / Core / Project
- Single Root Cause Audit

### 3. 降低清单页密度

文字密度最高的页面集中在第 37、38、47、11、23、40、27、10、32、39 等。建议主线页每页只保留一个主结论；清单页改为附录。

### 4. 统一“证据页”的呈现模式

建议证据页固定为：

- 左侧：问题 / 方法 / 结论
- 右侧：图或数据
- 底部：一句话 Takeaway

适用页面：

- 第 20 页
- 第 21 页
- 第 23 页
- 第 27 页
- 第 28 页

## 建议的最终结尾

当前结尾稍微偏工具与成果展陈。建议最后回到一个更强的观点：

> 这套流程真正改变的不是“谁写代码”，而是谁控制判断、边界、证据和知识回流。研究者从执行瓶颈，变成了判断中枢。

可以把这句话放在结尾页前一页，最后一页再保留 `END OF FIELDBOOK`。
